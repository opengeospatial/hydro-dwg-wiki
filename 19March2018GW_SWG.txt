---++ *Results*
   1 No change. Issue refers to an old version of GWML2 and is not applicable to the published spec.
   1 Specification change only: add explanatory text to indicate water wells can be traversed from boreholes using O&M.
      * Reconsideration of issue deferred pending developments in borehole standards harmonization.
   1 Changes to specification, schema, schematron rules, instance docs.
      * Remove gwml:result property between GW_GeologyLog and GW_GeologyLogCoverage and use om:result singularly for this relation.
         * No instance doc change or implementation changes required, as all known examples and implementations currently use om:result.
      * Add two compact encoding options for GW_GeologyLog result: SWE xml and block encodings
         * retain GW_GeologyLogCoverage as a valid option.
         * specification to indicate that it is a requirement (not recommendation) to use one of the three options
            * to provide expected options for software developers
            * communities can develop local profiles for other non-standard encoding options

   1 <span style="background-color: transparent;">Typo noticed in an element called "nomi</span><b style="background-color: transparent; font-size: 12.22px;"><u>c</u></b><span style="background-color: #f5f5f5; font-size: 12.22px;">alScreenDiameter" (with a "c") in a GWML Schema file </span> [[http://ngwd-bdnes.cits.nrcan.gc.ca/service/gwml/schemas/wellConstruction.xsd][here]]<span style="background-color: #f5f5f5; font-size: 12.22px;">. The description in the schema indicates that it is "the nominal screen diameter". The same element name and description appear in </span> [[http://gin.gw-info.net/service/api_ngwds:gin2/en/downloadmanager/dataset.zip?package=gwml&filter=gwml11\.pdf&t=1][the pdf]]<span style="background-color: #f5f5f5; font-size: 12.22px;"> describing the standard. Should the schema and/or documentation be updated so that the element is named "nominalScreenDiameter" (with an "n")?</span>
   1 <span style="background-color: transparent;"> Provide an association from Borehole to !WaterWell </span> http://ogc.standardstracker.org/show_request.cgi?id=508
      * While GW_WaterWell has an association (gwWellConstruction) to link to associated Borehole, there are no link back. Because they both are SF_SamplingFeatures, a link can be done using om:relatedSamplingFeature, but this is a different pattern and provide a different route (not consistant) . It must be pointed out that om:relatedSamplingFeature can also be used for WaterWell -&gt;Borehole.We either need a note in spec to advise how to user relatedSamplingFeature for Borehore-&gt;WaterWell (prescribed role name) or a new property for consistency.
   1 Change borehole log encoding : http://ogc.standardstracker.org/show_request.cgi?id=532
      * The proposed encoding is verbose and forces repetition of large blocks of XML. Need more compact encoding, easier to parse and also imposed indirectly so that records remain consistent along the log (in the other encoding, the DataRecord could technically change)
      * There was discussion about how critical the “serialisation” size really is; but encoding geophysics data can be extremely large.
      * Proposal:
         * Relax the requirement that “geology log” uses the GW_GeologyLogCoverage (but keep it as an option). This will allow to externalise the conformance to community on how they want to encode their data and provide core compact encoding, including swe:DataArray
